HOW IT WORKS:
=============

Your recording system now runs CONTINUOUSLY in 5-hour blocks:

Session 1: Runs for 5 hours → Completes
   ↓ (30 second gap)
Session 2: Automatically starts → Runs for 5 hours → Completes
   ↓ (30 second gap)
Session 3: Automatically starts → Runs for 5 hours → Completes
   ↓ (continues forever...)


WORKFLOW CONFIGURATION:
=======================

1. Main Workflow: continuous-recording.yml
   - Runs for 5 hours (timeout: 300 minutes)
   - Schedule: Every 5 hours (cron: '0 */5 * * *')
   - Triggers: Schedule, Manual, Push to main/master

2. Auto-Restart Workflow: auto-restart-recording.yml
   - Monitors main workflow completion
   - Waits 30 seconds after completion
   - Automatically triggers next session
   - Works for both success and failure


SCHEDULE TIMES (UTC):
=====================

With cron '0 */5 * * *', the workflow triggers at:
- 00:00 UTC
- 05:00 UTC
- 10:00 UTC
- 15:00 UTC
- 20:00 UTC

But with auto-restart, it runs continuously without waiting for schedule!


HOW AUTO-RESTART WORKS:
========================

1. Recording session runs for 5 hours
2. Session completes (success or failure)
3. Auto-restart workflow detects completion
4. Waits 30 seconds for cleanup
5. Triggers new recording session immediately
6. New session starts and runs for 5 hours
7. Repeat from step 2

Result: TRUE 24/7 CONTINUOUS RECORDING!


BENEFITS:

TROUBLESHOOTING:
================

If recording stops:
1. Check workflow runs: gh run list
2. Check for errors: gh run view [run-id]
3. Manually trigger: gh workflow run "continuous-recording.yml"
4. Check GitHub Actions tab in repository

If auto-restart fails:
1. Check auto-restart-recording.yml workflow
2. Verify GITHUB_TOKEN permissions
3. Check workflow_run trigger is working
